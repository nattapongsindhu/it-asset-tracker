ฉบับภาษาไทย

จัดทำเมื่อ 21 เมษายน 2569

คู่มือฉบับนี้ใช้เพื่อควบคุมการทำงานของโปรเจกต์ตั้งแต่เริ่มต้นจนจบ โดยเน้นการลดความผิดพลาดจากการข้ามขั้นตอน ช่วยให้กลับมาทำงานต่อได้ง่ายเมื่อหยุดกลางคัน และทำให้ Lead Tech กับ Project Manager เห็นภาพเดียวกันตลอดทั้งโครงการ

โครงสร้างทั้งหมดอ้างอิง workflow หลักที่ต้องทำตามอย่างมีวินัย ได้แก่ /spec, /plan, /build, /test, /review และ /ship

สำหรับโปรเจกต์นี้ เป้าหมายด้านการยืนยันตัวตนคือ Supabase Hybrid Auth: รองรับทั้ง Email + Password และ Magic Link สำหรับผู้ใช้ที่มีบัญชีอยู่แล้ว โดย credential ของ staff สำหรับ QA ให้ใช้ staff123

1. หลักการควบคุมงาน

ทุก phase ต้องตอบคำถาม 5 ข้อนี้ให้ได้ก่อนจะขยับไปขั้นถัดไป

  1. ตอนนี้เรากำลังอยู่ phase อะไร
  2. เป้าหมายของ phase นี้คืออะไร
  3. สิ่งที่ต้องส่งมอบก่อนออกจาก phase นี้มีอะไรบ้าง
  4. หลักฐานอะไรใช้ยืนยันว่า phase นี้ผ่านแล้ว
  5. ถ้าหยุดงานตอนนี้ คนถัดไปต้องรู้อะไรเพื่อทำต่อได้ทันที

2. เอกสารกลางที่ต้องอัปเดตเสมอ

  • Spec / PRD: ปัญหา ขอบเขต ข้อจำกัด และ acceptance criteria
  • Implementation Plan: task ที่เรียงตาม dependency
  • Decision Log: การตัดสินใจสำคัญด้านสถาปัตยกรรมและขอบเขต
  • Risk / Blocker Log: ความเสี่ยง สิ่งที่ติดขัด เจ้าของงาน และวันติดตาม
  • Progress Log: งานที่เสร็จ กำลังทำ และรออยู่
  • Verification Log: lint, build, test, QA, deployment check
  • Handoff Note: บริบทขั้นต่ำที่ต้องทิ้งไว้เมื่อหยุดงาน

3. ภาพรวมแต่ละ Phase

Phaseเป้าหมายหลักสิ่งที่ต้องส่งมอบGate ก่อนออกจาก Phase
/specนิยามงานให้ชัดTechnical spec หรือ PRDยังไม่เริ่ม implementation
/planทำ roadmap ที่เรียงตาม dependencyTask plan พร้อม definition of doneทุก task มี owner และ verification method
/buildพัฒนาเป็น slice เล็ก ๆCode และ config change ของ slice นั้นslice ยังต้องย้อนกลับได้
/testพิสูจน์ว่างานใช้ได้จริงหลักฐานจากคำสั่งและการตรวจสอบห้ามสรุปผลโดยไม่มี evidence
/reviewตรวจคุณภาพและความเสี่ยงFindings, fixes, residual risksห้ามมี critical issue ค้างแบบเงียบ ๆ
/shipปล่อยขึ้น production อย่างปลอดภัยDeployment checklist และ rollback planพร้อม post-release verification

4. สิ่งที่ต้องทำก่อนเข้า Phase แรก

  • ระบุ owner ของงาน
  • ระบุว่างานรอบนี้เป็น feature, migration หรือ bug fix
  • ระบุ stakeholder หลัก
  • ระบุ milestone หรือ deadline ถ้ามี
  • แยกสิ่งที่เป็น fact ออกจาก assumption ที่ยังต้องพิสูจน์

ก่อนเริ่ม Phase 1 ทุกคนควรรู้ตรงกันว่างานนี้คืออะไร และอะไรไม่อยู่ในรอบงานนี้

5. รายละเอียดแต่ละ Phase

5.1 /spec

เป้าหมาย: ตัดความคลุมเครือออกให้มากที่สุดก่อนเริ่มแก้โค้ด

  • วิเคราะห์คำขอและตรวจ repo ปัจจุบันก่อนเสมอ
  • ระบุ current state, target state, scope และ non-scope
  • ระบุ stack, constraints, acceptance criteria และความเสี่ยงหลัก
  • บันทึกคำถามที่ยังไม่ปิด
Stakeholderสิ่งที่ต้องเห็น
Lead Techทิศทาง architecture, business rule, migration risk, security impact
Project Managerความชัดเจนของ scope, ขอบเขต MVP, ความเสี่ยงต่อ timeline, dependency ภายนอก
  • ห้ามเขียน implementation code ใน phase นี้
  • ห้ามเดา requirement โดยไม่ตรวจ context จริง

5.2 /plan

เป้าหมาย: แปลง spec ให้เป็น roadmap ที่ track ได้จริง

  • แตกงานเป็น task เล็ก ๆ
  • เรียง task ตาม dependency
  • แยก task เสี่ยงออกจาก task ปกติ
  • ระบุ file, module, schema และ environment ที่ได้รับผลกระทบ
  • ใส่ definition of done และ verification method ให้ทุก task
Stakeholderสิ่งที่ต้องเห็น
Lead Techลำดับ technical task ที่ปลอดภัย และการแยกงานเสี่ยงออกจากกัน
Project Managermilestone, checkpoint, และภาพ dependency ของงาน
  • ห้ามรวมหลายธีมใหญ่ไว้ใน task เดียว
  • ห้ามเริ่ม build ถ้า plan ยังไม่ชัด

5.3 /build

เป้าหมาย: พัฒนาแบบ incremental โดยให้ verify และย้อนกลับได้

  • ลงมือทำทีละ slice
  • คุม scope ของ slice ให้เล็กและชัด
  • อัปเดต progress หลังจบแต่ละ slice
  • ถ้าเจอ risk ใหม่ที่กระทบ architecture ให้หยุดและปรับ plan
  • ทำ foundation ก่อน เช่น schema, auth, guards, env แล้วค่อย flow หลัก
Stakeholderสิ่งที่ต้องเห็น
Lead Techคุณภาพ implementation, ความสอดคล้องกับ plan, และการไม่ over-engineer
Project Managerslice ที่เสร็จแล้ว, slice ปัจจุบัน, blocker ใหม่ หรือความล่าช้า
  • ห้าม build หลาย slice ที่ verify ยากพร้อมกัน
  • ห้ามเปลี่ยน architecture กลางทางแบบไม่บันทึก

5.4 /test

เป้าหมาย: เปลี่ยน assumption ให้เป็น proof

  • รัน lint, build และ automated tests ที่มี
  • ทำ manual checks บน critical path
  • ตรวจ role-based access, regression risk และ flow สำคัญ
  • บันทึกคำสั่งที่รัน สถานะ pass/fail และช่องว่างของการทดสอบ
Stakeholderสิ่งที่ต้องเห็น
Lead Techcoverage ของ path เสี่ยง เช่น auth, data, security, และ critical workflow
Project Managerความพร้อมเข้า review, defect ที่ยังเปิดอยู่, และ dependency ฝั่ง QA
  • ห้ามบอกว่างานเสร็จโดยไม่มี evidence
  • ห้ามซ่อน failure หรือข้าม critical-path verification

5.5 /review

เป้าหมาย: ตรวจ implementation ให้พร้อมใช้งานจริงในมุม readability, security, maintainability และ production safety

  • review readability และความเรียบง่ายของโค้ด
  • review authorization, data protection และ security gap
  • review maintainability, duplication และ abstraction ที่เกินจำเป็น
  • review documentation, onboarding และ operational readiness
Stakeholderสิ่งที่ต้องเห็น
Lead Techcritical issue, code smell, maintainability concern และ readiness ก่อน release
Project Managerresidual risk, decision ที่ต้อง approve และ release blocker ที่ยังเหลือ
  • ห้ามลด review ให้เหลือแค่เรื่อง formatting
  • ห้ามปล่อย security gap ที่รู้แล้วโดยไม่บันทึก

5.6 /ship

เป้าหมาย: ปล่อยขึ้น production อย่างปลอดภัยและรู้เส้นทาง rollback ชัดเจน

  • ตรวจ env vars, secrets, schema readiness และ deployment config
  • เตรียม release note และ rollback instruction
  • ทำ pre-release และ post-release verification
  • ระบุ monitoring หรือ log ที่ต้องจับตาหลัง release
Stakeholderสิ่งที่ต้องเห็น
Lead Techdeployment risk, rollback path และแผน monitoring หลังปล่อย
Project Managerrelease window, stakeholder communication และสถานะหลังปล่อย
  • ห้ามปล่อยโดยไม่มี rollback plan
  • ห้ามถือว่า deploy ผ่านเท่ากับ release สำเร็จ

6. กฎเมื่อหยุดงานและต้องกลับมาทำต่อ

ทุกครั้งที่หยุดงานกลางคัน ต้องทิ้ง handoff note อย่างน้อย 8 รายการนี้ไว้เสมอ

  1. Current Phase
  2. Objective ของ phase ปัจจุบัน
  3. งานที่ทำเสร็จแล้ว
  4. งานที่ยังไม่เสร็จ
  5. Blocker หรือ Risk ล่าสุด
  6. Decision สำคัญล่าสุด
  7. Evidence ที่มีแล้ว
  8. Next Step ที่ควรทำต่อทันที

6.1 ตัวอย่าง Handoff Note

Current Phase: /build. Current Slice: Hybrid auth guard update. Done: server auth helper, route guard, session role mapping และเส้นทาง login แบบ password. Not Done: การ verify callback ของ magic link และ staff restriction check บน /audit. Blocker: ต้องการ final decision เรื่อง scope การมองเห็น asset ของ staff. Latest Decision: ใช้ Supabase Hybrid Auth เป็น target และไม่ใช้ Prisma bridge สำหรับ auth. Evidence: lint ผ่านแล้ว, manual login test ยังไม่รัน. Next Step: ปิด magic-link authorization check และ /audit guard แล้วค่อยเข้า /test.

7. รูปแบบรายงานสถานะสำหรับ Lead Tech และ PM

ควรใช้ format เดิมตลอดโปรเจกต์เพื่อให้ทุกคนอ่าน progress จากมุมเดียวกัน

  • Current Phase
  • Completed This Period
  • In Progress
  • Next Phase Gate
  • Risks / Blockers
  • Decisions Needed
  • ETA to Next Milestone

8. Checklist คุมคุณภาพระดับโปรเจกต์

  • ทุก phase มี owner ชัดเจน
  • ทุก phase มี exit criteria
  • ทุก phase มี evidence
  • ทุกครั้งที่หยุดงานมี handoff note
  • ทุก decision สำคัญถูกบันทึก
  • ทุก blocker มี owner และวัน follow-up
  • ทุก release มี rollback plan

9. สรุปสั้น

  • Spec ก่อน Build
  • Build ทีละ Slice
  • ทุก Slice ต้อง Verify
  • Review ก่อน Ship
  • หยุดเมื่อไรต้องทิ้ง Handoff Note

เมื่อทำตามกฎนี้ โปรเจกต์จะเดินต่อได้แม้เปลี่ยนคนทำ หยุดงานกลางคัน หรือมีหลาย stakeholder ต้องติดตามงานพร้อมกันตั้งแต่ kickoff จน release
